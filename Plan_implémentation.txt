# Plan d'implementation revise - Gestion Stock Vision

## Objectif du prototype

Construire une chaine de demonstration pour une pharmacie ou un stock medical:

1. recevoir une demande d'article;
2. verifier le stock;
3. selectionner l'image associee au produit;
4. simuler le deplacement du robot vers le bon rayon;
5. preparer le branchement futur avec la vision par ordinateur et l'automatisation.

Le travail demande pour cette part d'equipe concerne l'etape 1: modules 1 et 2.

## Etape 1 - Modules 1 et 2: implementee

### Module 1: gestion de stock

Fichiers crees:

- `data/inventory.csv`: inventaire initial de 8 articles.
- `app.py`: lecture du CSV, recherche par ID ou nom, controle de quantite, messages d'erreur.
- `gui.py`: interface CustomTkinter pour le poste pharmacien.
- `tests/test_stock_simulation.py`: tests des scenarios principaux.

Attributs suivis pour chaque article:

- `id_unique`
- `nom`
- `classe_therapeutique`
- `emplacement_rayon`
- `quantite_stock`
- `prix`
- `image_path`

Scenarios deja couverts:

- article trouve par ID;
- article trouve par nom partiel unique;
- article introuvable;
- quantite invalide;
- stock insuffisant;
- image associee manquante.

### Module 2: selection et simulation d'image

Fichiers crees:

- `images/*.svg`: images simulees pour chaque article.
- la colonne `image_path` du CSV associe chaque article a son image.

Comportement actuel:

- si l'article existe et que le stock suffit, le programme affiche l'image selectionnee;
- il simule le mouvement avec le message:
  `Robot allant au compartiment: {emplacement_rayon}`;
- il s'arrete avant la vision par ordinateur, comme prevu pour cette etape.

Commandes utiles:

```powershell
python -m pip install -r requirements.txt
python gui.py
python app.py --list
python app.py --query MED-001 --quantity 2
python app.py --query amoxicilline
python -m unittest discover -s tests
```

### Interface graphique retenue

CustomTkinter est retenu pour la version locale:

- panneau gauche: inventaire et selection rapide d'article;
- panneau central: demande client, quantite et simulation;
- panneau droit: resultat, image selectionnee et point d'appel vision;
- zone basse: brouillon de contexte LLM pour assister le pharmacien.

Le LLM ne doit pas delivrer automatiquement un medicament. Il doit aider a
poser des questions de clarification, resumer les symptomes et laisser la
validation au pharmacien.

## Etape 2 - Vision par ordinateur: ajustement important

Le plan initial disait que `yolo11n.pt` etait pret pour detecter des pilules. C'est a corriger: les poids YOLO11 fournis par Ultralytics sont principalement des modeles generiques preentraines sur COCO pour la detection, et COCO ne correspond pas a un jeu de classes medicales dedie aux comprimes.

Plan recommande pour la suite:

1. garder `app.py` comme point d'entree du flux stock -> image;
2. remplacer le placeholder `process_image(image_path)` dans `vision.py`;
3. utiliser soit un modele de detection entraine sur des images de medicaments, soit un petit dataset annote pour fine-tuner un modele YOLO;
4. retourner un verdict simple:
   - `VISION_OK`: pilule detectee;
   - `VISION_KO`: pilule non detectee ou image invalide;
5. ne decrementer le stock qu'apres confirmation de la vision.

## Etape 3 - Automatisation

A implementer apres validation de la vision:

- decrementer le stock dans une copie ou une base persistante;
- journaliser les transactions dans `transactions.csv`;
- generer des instructions de livraison;
- conserver les erreurs dans un fichier de logs pour la demonstration.

## Etape 4 - Interface, LLM et demo

Options possibles:

- CustomTkinter pour la demo locale actuelle;
- Streamlit si l'equipe veut une demo web rapide;
- Flask/FastAPI si l'equipe veut exposer une API.

Pour une presentation courte locale, CustomTkinter suffit: inventaire visible,
formulaire de demande, image selectionnee, statut vision et brouillon LLM.

## Priorites immediates

1. Valider l'etape 1 avec l'equipe, y compris l'interface CustomTkinter.
2. Brancher d'abord la vision dans `vision.py`.
3. Remplacer les images SVG simulees par de vraies photos libres ou par un mini-dataset annote.
4. Choisir le modele de vision et verifier ses classes.
5. Ajouter ensuite l'automatisation seulement quand la verification image donne un resultat fiable.
